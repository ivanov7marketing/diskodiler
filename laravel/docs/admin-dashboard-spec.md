# Минимальный дашборд админки

Дата: 21 апреля 2026

## 1. Цель

Сделать один понятный экран в админке, который за 10-15 секунд отвечает на три вопроса:

- сколько заявок пришло за ключевые периоды;
- из каких страниц и каналов приходят лиды;
- в каком состоянии находится каталог прямо сейчас.

Этот дашборд пока не делится по ролям, не включает Topvisor и не пытается заменить полноценную BI-аналитику. Его задача — дать компактную управленческую сводку внутри Filament.

## 2. Состав дашборда

В минимальную версию входят 5 блоков:

- карточки лидов за `сегодня / 7 дней / 30 дней`;
- воронка статусов `новые -> в работе -> связались -> закрыты`;
- топ страниц, которые дают заявки;
- источники лидов `SEO / прямой переход / Директ / другие`;
- состояние каталога `всего / активные / без фото / без цены / без SEO / без видео`.

## 3. Предлагаемая компоновка экрана

### 3.1. Десктоп

| Ряд | Блок | Ширина |
| --- | --- | --- |
| 1 | Лиды за сегодня | 4/12 |
| 1 | Лиды за 7 дней | 4/12 |
| 1 | Лиды за 30 дней | 4/12 |
| 2 | Воронка лидов | 8/12 |
| 2 | Источники лидов | 4/12 |
| 3 | Топ страниц по лидам | 7/12 |
| 3 | Состояние каталога | 5/12 |

### 3.2. Мобильная версия

Блоки идут друг под другом в таком порядке:

- карточки лидов;
- воронка;
- источники;
- топ страниц;
- состояние каталога.

### 3.3. Общий визуальный принцип

- Верхний ряд должен быть самым спокойным и читаемым: крупная цифра, короткий лейбл, маленькая подпись.
- Второй ряд должен давать “историю” по лидам: что пришло и откуда.
- Третий ряд должен отвечать на вопрос “что работает” и “что сломано/не заполнено”.
- Между карточками нужен заметный воздух, чтобы экран не выглядел как перегруженная таблица.

## 4. Подробное описание блоков

### 4.1. Карточки лидов за периоды

Назначение: показать объём входящих заявок в трёх понятных временных окнах без дополнительных фильтров.

Состав:

- `Лиды сегодня`
- `Лиды за 7 дней`
- `Лиды за 30 дней`

Правило расчёта:

- считается по `leads.created_at`;
- используется таймзона приложения;
- `сегодня` считается от `00:00` текущего дня до текущего времени;
- `7 дней` включает текущий день и 6 предыдущих;
- `30 дней` включает текущий день и 29 предыдущих.

Что показываем внутри карточки:

- крупное число по центру или слева;
- короткий лейбл;
- маленькую подпись с периодом;
- иконку календаря или графика;
- кликабельность в список заявок с уже применённым фильтром по периоду.

Визуально:

- все 3 карточки одинаковой высоты;
- белый фон карточек;
- мягкая тень;
- скругление крупнее стандартного;
- акцентный цвет можно оставить в палитре Filament `amber`, чтобы дашборд не выпадал из текущей темы.

### 4.2. Воронка лидов

Назначение: быстро показать, что происходит с заявками после попадания в систему.

Этапы воронки:

- `Новые` = `status = new`
- `В работе` = `status = in_progress`
- `Связались` = `status = contacted`
- `Закрыты` = `status = done`

Отдельное правило:

- статус `no_answer` не включается в основную воронку, чтобы она оставалась простой и линейной;
- `no_answer` можно показывать маленьким вторичным бейджем в правом верхнем углу блока как дополнительную служебную цифру.

Период:

- по умолчанию показываем данные за `30 дней`;
- в шапке блока делаем компактный переключатель `7 дней / 30 дней`;
- этот же переключатель должен влиять и на блоки `Источники` и `Топ страниц`, чтобы все три аналитических блока сравнивались в одном окне.

Что показываем:

- число лидов на каждом этапе;
- процент от первого шага;
- тонкую подпись под шагом;
- визуальную связь между шагами.

Рекомендуемый вид:

- не обычная таблица, а горизонтальная последовательность из 4 карточек-шагов;
- внутри шага крупная цифра, ниже название статуса, ещё ниже процент;
- между шагами лёгкие соединители или стрелки;
- цвета должны быть спокойными: жёлтый для новых, синий для работы, зелёный для контакта, серый для закрытых.

### 4.3. Топ страниц, которые дают заявки

Назначение: показать, какие страницы реально приводят лиды, а не просто существуют в каталоге.

Период:

- использует общий переключатель `7 дней / 30 дней`;
- по умолчанию стоит `30 дней`.

Источник данных:

- базовое поле: `leads.source_page`.

Перед расчётом нужна нормализация `source_page`:

- убрать домен, если он попал в поле целиком;
- убрать query string и hash;
- привести `/path/` и `/path` к одному виду;
- пустое значение группировать как `Не определено`;
- главную страницу хранить как `/`.

Что показываем:

- топ-7 страниц по количеству лидов;
- путь страницы;
- число заявок;
- долю от общего числа лидов за выбранный период.

Формат блока:

- компактная таблица или список с полосой прогресса;
- первая колонка — путь страницы;
- вторая колонка — число лидов;
- третья — доля в процентах;
- слишком длинные URL обрезаются, но целиком доступны в `tooltip`.

Польза:

- видно, какие товарные или сервисные страницы действительно работают;
- видно, где стоит усиливать контент, CTA и формы.

### 4.4. Источники лидов

Назначение: быстро понять структуру трафика без ухода во внешние системы.

Категории:

- `SEO`
- `Прямой переход`
- `Директ`
- `Другие`

Важное замечание по данным:

- в текущем виде в лидах уже есть `utm` и `source_page`;
- этого недостаточно, чтобы надёжно разделять `SEO` и `прямой переход`;
- для честной классификации нужно добавить в лиды ещё минимум `referer_url`, `referer_host` и вычисляемое поле `traffic_channel`.

Рекомендуемая минимальная схема для источников:

| Поле | Статус |
| --- | --- |
| `utm` | уже есть |
| `source_page` | уже есть |
| `referer_url` | нужно добавить |
| `referer_host` | нужно добавить |
| `traffic_channel` | нужно добавить |

Правила определения `traffic_channel`:

| Канал | Правило |
| --- | --- |
| `Директ` | есть признаки платного трафика из Яндекс Директа: `utm_source=yandex`, `utm_medium=cpc`, `utm_campaign` или другая метка с явным paid/direct-паттерном |
| `SEO` | нет признаков paid-трафика, а `referer_host` указывает на поисковую систему |
| `Прямой переход` | нет UTM и нет внешнего referer |
| `Другие` | всё, что не попало в три категории выше |

Период:

- использует общий переключатель `7 дней / 30 дней`;
- по умолчанию стоит `30 дней`.

Формат блока:

- donut chart на 4 сегмента;
- справа компактная легенда со значением и процентом;
- цвета должны быть различимы, но без кислотности.

Почему это важно:

- руководству видно, растёт ли доля органики;
- видно, не завязаны ли лиды только на платный канал;
- можно принимать решение по приоритету SEO-работ без захода в Метрику.

### 4.5. Состояние каталога

Назначение: показать не продажи, а готовность каталога к нормальной работе и продвижению.

Состав метрик:

- `Всего товаров`
- `Активные товары`
- `Без фото`
- `Без цены`
- `Без SEO`
- `Без видео`

Правило расчёта:

- `Всего товаров` = все записи `products`;
- `Активные товары` = `products.active = true`;
- проблемные метрики `без фото / без цены / без SEO / без видео` считаются только по активным товарам;
- это нужно, чтобы дашборд отражал состояние публичного каталога, а не черновиков.

Точные правила:

| Метрика | Логика |
| --- | --- |
| `Всего товаров` | общее количество товаров |
| `Активные товары` | `active = true` |
| `Без фото` | активные товары без связанных `images` |
| `Без цены` | активные товары, у которых `price` пустой и `price_label` пустой |
| `Без SEO` | активные товары, у которых пустой `meta_title` или пустой `meta_description` |
| `Без видео` | активные товары, у которых одновременно пустые `video_path` и `video_url` |

Формат блока:

- одна большая карточка;
- внутри сетка `2 x 3` на десктопе;
- каждая метрика — отдельная плитка с крупным числом и коротким лейблом;
- проблемные метрики должны быть кликабельны и вести в список товаров с нужным фильтром.

Визуально:

- `Всего` и `Активные` можно делать нейтрально-зелёными;
- проблемные плитки должны иметь мягкий warning/danger-акцент;
- плитки должны быть компактными, чтобы весь блок читался как одна карта здоровья каталога.

## 5. Фильтры и взаимодействия

В минимальной версии достаточно двух уровней поведения:

- карточки `сегодня / 7 дней / 30 дней` всегда фиксированы и не зависят от общего фильтра;
- блоки `Воронка`, `Источники`, `Топ страниц` используют общий переключатель периода `7 дней / 30 дней`.

Дополнительно:

- каждый блок должен быть кликабельным в соответствующий раздел админки;
- внизу или в правом верхнем углу страницы полезно вывести `Данные обновлены: дата/время`;
- если данных нет, вместо пустого графика показывается аккуратное состояние `Пока нет данных`.

## 6. Требования к данным перед реализацией

Для запуска дашборда без Topvisor и без внешних интеграций уже хватает текущих данных по товарам и части данных по лидам. Но для блока `Источники` нужно минимально расширить модель лидов.

Минимально нужно:

- сохранять `referer_url` из запроса;
- отдельно сохранять `referer_host`;
- при создании лида вычислять `traffic_channel`;
- нормализовать `source_page` до пути страницы;
- по возможности добавить индекс на `created_at` и `status`, если объём лидов начнёт расти.

Без этих полей блок `Источники` будет визуально красивым, но логически неточным.

## 7. Требования к внешнему виду

Дашборд должен быть не просто “набором виджетов”, а цельным экраном.

Ключевые требования:

- чистый светлый фон страницы;
- одинаковые отступы между карточками;
- одинаковая логика заголовков;
- крупные цифры без визуального шума;
- минимальное количество линий и рамок;
- один акцентный брендовый цвет плюс спокойные вспомогательные цвета статусов;
- никакой тяжёлой псевдо-аналитики ради красоты.

Практический стиль:

- заголовок страницы `Дашборд`;
- подзаголовок одной строкой, например: `Лиды, источники и состояние каталога в одном экране`;
- карточки со скруглением 18-24 px;
- внутренний padding 20-24 px;
- мягкие тени;
- иконки только там, где они помогают навигации.

## 8. Техническая реализация в Filament

Для реализации лучше использовать не стандартный пустой `Dashboard`, а свой набор виджетов.

Рекомендуемый набор:

- `LeadPeriodStatsWidget`
- `LeadFunnelWidget`
- `LeadSourcesWidget`
- `TopLeadPagesWidget`
- `CatalogHealthWidget`

Дополнительно:

- убрать из первого экрана стандартный `FilamentInfoWidget`;
- `AccountWidget` можно оставить, если он не мешает композиции;
- агрегаты лучше кэшировать на 5 минут, чтобы экран открывался быстро.

## 9. Что считаем итогом для версии v1

Версия `v1` считается удачной, если:

- экран открывается и сразу даёт понимание по лидам;
- видно, какие страницы реально приводят заявки;
- видно, как распределяются источники;
- видно, в каком состоянии активный каталог;
- по каждой проблемной цифре можно перейти в соответствующий список и сразу работать дальше.

Если это выполнено, дашборд уже полезен и без Topvisor, и без сложного деления по ролям.
