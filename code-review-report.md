# Отчет по анализу кода проекта

Дата проверки: 18 апреля 2026

Проект выглядит как аккуратный статический сайт/прототип: несколько HTML-страниц, общий CSS и один общий JS-файл. `assets/js/app.js` проходит синтаксическую проверку `node --check`, git-дерево на момент проверки было чистым. Для небольшого лендинга структура рабочая, но для полноценного коммерческого сайта есть важные проблемы.

## Findings

### P1. Формы создают иллюзию отправки, но заявки никуда не уходят

Файлы:

- `assets/js/app.js:104`
- `assets/js/app.js:123`
- `assets/js/app.js:299`
- `assets/js/app.js:127`

В обработчиках форм используется `event.preventDefault()`, после чего сайт просто показывает сообщение об успехе. Реальной отправки в CRM, на сервер, почту или мессенджер нет.

В `assets/js/app.js:127` прямо указано: "Для демонстрации сайта отправка подключается к CRM на этапе интеграции".

Риск: пользователь думает, что заявка принята, но бизнес фактически теряет лиды.

Рекомендация: подключить реальную отправку заявок: CRM webhook, backend endpoint, email API или хотя бы отправку в Telegram/WhatsApp через понятный сценарий.

### P1. Кнопка "Добавить в корзину" ничего не делает

Файл:

- `product.html:71`

На странице товара кнопка `Добавить в корзину` имеет `type="button"` и только `data-goal`. Обработчика корзины нет.

Риск: это кликабельный тупик на важном коммерческом действии.

Рекомендация: если корзина не планируется, заменить кнопку на более честное действие: "Забронировать", "Написать в WhatsApp", "Оставить заявку" или открытие контактной модалки.

### P2. Все карточки каталога ведут на один и тот же товар

Файл:

- `assets/js/app.js:444`

В каталоге каждая карточка получает ссылку `href="product.html"`. При этом в данных каталога есть разные товары BMW, Mercedes-Benz, Porsche и Range Rover, но клик "Подробнее" всегда открывает один демо-товар `19" Style 793 Individual`.

Риск: пользователь выбирает один товар, а попадает на другой. Это снижает доверие и ломает сценарий каталога.

Рекомендация: добавить slug/url для каждого товара и либо сделать отдельные страницы товаров, либо передавать id/slug в query string и динамически рендерить карточку товара.

### P2. Нет нормальной валидации телефона и лид-форм

Файлы:

- `assets/js/app.js:107`
- `services.html:72`
- `index.html:342`
- `catalog.html:86`

VIN-форма проверяет только длину VIN. Телефон в VIN-формах и lead-формах фактически может быть пустым. Сервисная форма и exit-popup могут показать успешную отправку без телефона.

Риск: заявки могут быть непригодны для обработки.

Рекомендация: добавить `required` для телефона, нормализацию номера, базовую проверку длины и понятные сообщения об ошибках.

### P2. Повторяется разметка шапки, футера, модалок и плавающих кнопок — решено

Файлы:

- `index.html:46`
- `catalog.html:25`
- `product.html:38`
- `services.html:13`
- `about.html:35`

Одинаковые блоки продублированы на всех страницах. Уже видны расхождения в мобильных меню, `aria-label`, модалках и CTA.

Риск: любое изменение шапки, футера или контактов нужно вручную повторять в нескольких файлах. Это повышает шанс регрессий.

Рекомендация: вынести общие блоки в шаблоны или перейти на простой сборщик/генератор статических страниц. Минимальный вариант: хранить общие фрагменты в одном месте и генерировать HTML перед публикацией.

Статус от 18 апреля 2026: решено без добавления сборщика. Общая шапка, футер, плавающие кнопки, VIN/contact/service-модалки и exit-popup теперь рендерятся из `assets/js/app.js`, а HTML-страницы содержат только `data-page` и `data-shared-*` placeholders.

### P3. `localStorage` читается без защиты и затем рендерится через `innerHTML`

Файлы:

- `assets/js/app.js:524`
- `assets/js/app.js:529`
- `assets/js/app.js:530`

`JSON.parse(localStorage.getItem(...))` может упасть, если значение повреждено. Затем данные из `localStorage` вставляются через `innerHTML`.

Риск небольшой, потому что данные в основном пишет сам сайт, но код хрупкий и небезопасный по привычке.

Рекомендация: обернуть чтение в `try/catch`, валидировать поля и собирать DOM через `createElement` или экранировать значения.

### P3. Много inline-стилей и данных прямо в HTML/JS

Примеры:

- `product.html:71`
- `product.html:104`
- `services.html:56`
- `index.html:192`
- `assets/js/app.js:318`

Часть визуальных отступов и фоновых изображений задана inline-стилями. Каталог товаров хранится прямо в JS.

Риск: поддержка усложняется по мере роста сайта.

Рекомендация: перенести inline-стили в CSS-классы, а данные каталога вынести в JSON или другой единый источник данных.

## Что хорошо

- CSS организован достаточно аккуратно: есть CSS-переменные, адаптивные брейкпоинты, стабильные карточки, `aspect-ratio`, `min-width: 0`, `overflow-wrap`.
- JS разбит на понятные init-функции.
- Есть защита от повторной инициализации части обработчиков через `dataset`.
- Квиз реализован без грубых синтаксических ошибок.
- Иконки мессенджеров подключены локально.
- `assets/js/app.js` проходит `node --check`.

## Итог

Текущий проект больше похож на качественный статический прототип/лендинг, чем на полностью готовый коммерческий сайт.

Главные доработки перед продакшеном:

1. Подключить реальную отправку форм.
2. Убрать или оживить неработающие коммерческие CTA.
3. Сделать корректные страницы или динамическую деталку товаров.
4. Добавить нормальную валидацию телефона и заявок.
5. Уменьшить дублирование общих HTML-блоков.

## Проверка

Выполнено:

```powershell
node --check assets\js\app.js
```

Синтаксических ошибок в JS не найдено.

Не выполнялось:

- визуальный прогон в браузере;
- проверка HTML-валидатором;
- Lighthouse/PageSpeed;
- проверка отправки форм, так как backend/CRM-интеграции в проекте нет.
